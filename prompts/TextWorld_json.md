# Аналіз JSON-файлу TextWorld

## Огляд проекту

**TextWorld** — це розроблена Microsoft бібліотека для створення та запуску текстових пригодницьких ігор. Вона призначена для:
- навчання агентів штучного інтелекту (reinforcement learning);
- процедурної генерації текстових ігор;
- досліджень у галузі розуміння природної мови.

### Команда `tw-make`

Команда `tw-make` генерує текстові пригодницькі ігри з шаблонів:

```bash
tw-make tw-simple --goal brief --seed 5 --output simple_game.z8 --rewards sparse
```

**Параметри:**
- `tw-simple` — тип шаблону гри;
- `--goal brief` — стиль опису мети квесту;
- `--seed 5` — випадкове зерно для процедурної генерації;
- `--output simple_game.z8` — вихідний файл гри (формат Z-machine);
- `--rewards sparse` — тип системи винагород.

### Згенеровані файли

Після створення гри `tw-make` формує:
1. **`.z8` файл** — скомпільована гра для інтерпретатора Z-machine;
2. **`.json` файл** — метадані та структура гри;
3. **`.ulx` файл** — формат Glulx (опціонально).

## Структура JSON-файлу

JSON-файл містить повну інформацію про стан гри:

### Структура верхнього рівня

| Поле | Тип | Опис |
|------|-----|------|
| `version` | int | Версія формату JSON (1) |
| `world` | масив | Предикати початкового стану світу |
| `grammar` | об'єкт | Налаштування граматики та обмеження гри |
| `quests` | масив | Визначення квестів з умовами перемоги/поразки |
| `infos` | масив | База даних інформації про об'єкти/сутності |

### Предикати світу (`world`)

Масив фактів, що визначають **початковий стан гри**:

| Предикат | Аргументи | Значення |
|----------|-----------|----------|
| `at` | `{сутність, кімната}` | Сутність розташована в кімнаті |
| `closed` | `{контейнер}` | Контейнер/двері зачинені |
| `locked` | `{двері}` | Двері замкнені |
| `open` | `{контейнер}` | Контейнер відчинений |
| `edible` | `{їжа}` | предмет можна з'їсти |
| `in` | `{предмет, контейнер}` | предмет всередині контейнера |
| `on` | `{предмет, поверхня}` | предмет на поверхні |
| `free` | `{кімната1, кімната2}` | прохід між кімнатами вільний |
| `link` | `{кімната1, двері, кімната2}` | двері з'єднують дві кімнати |
| `match` | `{ключ, двері}` | ключ підходить до дверей |
| `east_of`, `west_of`, `north_of`, `south_of` | `{кімната1, кімната2}` | напрямкове відношення |

### Типи сутностей

| Код типу | Значення | Приклади |
|----------|----------|----------|
| `P` | Гравець | P |
| `I` | Інвентар (предмети при собі) | I |
| `r` | Кімната | r_0 (спальня), r_1 (кухня) |
| `c` | Контейнер | c_0 (шухляда), c_2 (холодильник) |
| `d` | Двері | d_0 (дерев'яні двері), d_1 (двері з сіткою) |
| `f` | Їжа | f_1 (яблуко) |
| `k` | Ключ | k_0 (старий ключ) |
| `s` | Поверхня (supporter) | s_2 (плита) |
| `o` | Інший об'єкт | o_0 (записка) |

### Налаштування граматики

| Поле | Значення | Опис |
|------|----------|------|
| `theme` | "house" | тема гри |
| `names_to_exclude` | масив | Виключені назви об'єктів |
| `include_adj` | bool | Чи включати прикметники |
| `blend_descriptions` | bool | Об'єднувати описи кімнат |
| `ambiguous_instructions` | bool | Дозволити неоднозначні інструкції |
| `only_last_action` | bool | показувати лише результат останньої дії |
| `blend_instructions` | bool | Об'єднувати текст інструкцій |
| `allowed_variables_numbering` | bool | Дозволити нумерацію змінних |
| `unique_expansion` | bool | Вимагати унікальне розширення |

### Структура квестів

Кожен квест містить:

| Поле | Тип | Опис |
|------|-----|------|
| `desc` | рядок | Опис квесту |
| `reward` | int | бали винагороди за виконання |
| `commands` | масив | Потрібна послідовність команд |
| `win_events` | масив | події, що запускають перемогу |
| `fail_events` | масив | події, що призводять до поразки |
| `optional` | bool | Чи є квест опціональним |
| `repeatable` | bool | Чи можна повторити квест |

### події перемоги/поразки

| Поле | Тип | Опис |
|------|-----|------|
| `commands` | масив | Команди, що запускають подію |
| `actions` | масив | Визначення дій (для win_events) |
| `condition` | об'єкт | Умова, що запускає (для fail_events) |

### Структура дій

Кожна дія у `win_events[].actions[]`:

| Поле | Тип | Опис |
|------|-----|------|
| `name` | рядок | ідентифікатор дії (напр. "open/c", "take/c") |
| `preconditions` | масив | Предикати, що мають бути істинними для виконання |
| `postconditions` | масив | Предикати, що стануть істинними після виконання |
| `command_template` | рядок | Формат команди зі змінними |
| `reverse_name` | рядок | Назва зворотної дії |
| `reverse_command_template` | рядок | Формат зворотної команди |

### Структура infos

Масив пар `[id, object_info]`:

| Поле | Тип | Опис |
|------|-----|------|
| `id` | рядок | Унікальний ідентифікатор |
| `type` | рядок | тип сутності (P, I, r, c, d, f, k, s, o) |
| `name` | рядок/null | Зрозуміла назва |
| `noun` | рядок/null | базальне ім'я |
| `adj` | рядок/null | Прикметник |
| `desc` | рядок/null | текст опису (з умовами Inform7) |
| `room_type` | рядок/null | тип кімнати (storage, work, cook, clean) |
| `definite` | рядок/null | Визначений артикль |
| `indefinite` | рядок/null | Неозначений артикль |
| `synonyms` | масив/null | Альтернативні назви |

## Типи дій у цій грі

| дія | Опис | Приклад |
|--------|-------------|---------|
| `open/c` | Відчинити контейнер | "open chest drawer" |
| `close/c` | Зачинити контейнер | "close chest drawer" |
| `open/d` | Відчинити двері | "open wooden door" |
| `close/d` | Зачинити двері | "close wooden door" |
| `unlock/d` | Відчинити двері ключем | "unlock wooden door with old key" |
| `lock/d` | Замкнути двері ключем | "lock wooden door with old key" |
| `take/c` | Взяти предмет з контейнера | "take old key from chest drawer" |
| `take/s` | Взяти предмет з поверхні | "take apple from stove" |
| `insert` | покласти предмет у контейнер | "insert old key into chest drawer" |
| `put` | покласти предмет на поверхню | "put apple on stove" |
| `go/east` | Рух на схід | "go east" |
| `go/west` | Рух на захід | "go west" |
| `trigger` | Автоматична подія | (внутрішня подія) |

## Алгоритм роботи `tw-make`

```
1. Розбір аргументів командного рядка
2. Вибір шаблону гри (tw-simple)
3. Встановлення випадкового зерна (5)
4. Генерація ігрового світу:
   а. Створення кімнат зі з'єднаннями
   б. Розміщення контейнерів та поверхонь у кімнатах
   в. Розміщення об'єктів (ключів, їжі тощо) у контейнерах/на поверхнях
   г. Налаштування дверей та замків
5. Генерація квестів:
   а. Визначення мети на основі прапора --goal
   б. Створення потрібної послідовності дій
   в. Визначення умов перемоги/поразки
6. Компіляція гри у формат Z-machine (.z8)
7. Експорт файлу метаданих JSON
```

## Ключові спостереження

1. **Система на основі станів**: усе представлено як предикати (факти)
2. **дія = перехід стану**: дії перетворюють передумови → наслідки
3. **Квести = послідовність дій**: перемога, коли всі дії виконані по порядку
4. **Умови поразки**: явні fail_events можуть перервати квест
5. **двонаправлені дії**: більшість дій мають зворотну (open/close, take/insert)
6. **Шаблони змінних**: команди використовують синтаксис `{змінна}` для інстанціювання
